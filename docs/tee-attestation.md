# Attestation

Wattz treats compute honesty as a first-class concern. During the bootstrap
phase inference is relayed through Groq LPU capacity; responses on that path
report `attestation.kind = "relay"` and `attestation.verified = false`. Once a
bare-metal node serves a request, its response carries an attestation envelope
that the gateway verifies with `packages/compute-verifier` before the receipt
is trusted.

## Scope of `packages/compute-verifier`

The crate parses the binary layout prescribed by each vendor specification and
checks the embedded ECDSA / EdDSA signature against a per-node key pinned at
registration. Full trust-chain verification against the platform provisioning
certificates (Intel PCS collateral, AMD ARK / ASK, NVIDIA NRAS root) is on the
roadmap and is deliberately out of scope of this crate today: the gateway pins
the trusted attestation key after an out-of-band bootstrap flow, and the
verifier checks signatures against that pinned key.

All parsing routines are hardened against short inputs and reject reports that
violate the canonical length constraints in the vendor specifications.

## Supported Envelope Types

### Intel SGX (DCAP)

- Quote generated by the enclave inside `node-runtime` and sent in the
  `x-wattz-attestation` envelope.
- Verifier: `packages/compute-verifier::tee::sgx` (P-256 ECDSA).

### AMD SEV-SNP

- Attestation report produced by the SEV-SNP firmware.
- Verifier: `packages/compute-verifier::tee::sev` (P-384 ECDSA). The caller is
  responsible for validating the VCEK chain against the AMD ARK / ASK roots
  before pinning the key.

### NVIDIA Confidential Computing (H100 CC / H200 CC)

- Per-response attestation blob from `nvattest`.
- Verifier: `packages/compute-verifier::tee::nvidia_cc`.

Unknown envelope types are rejected rather than accepted optimistically.

## Optional ZK Compute Proofs

For a subset of small models, a node may attach a Risc0 or SP1 receipt of the
inference computation, verified via `packages/compute-verifier::zk::{risc0,
sp1}` by checking the outer EdDSA commitment over the zkVM output digest. When
present, the receipt PDA records the proof hash, and the dispute path
(`open_dispute` + `resolve_dispute`) can require it before resolving.

## Verification Timing

- Verification runs on the gateway host, not on the client. Clients receive
  only the summary claim: `kind` and `verified`, plus the node identity in the
  `x-wattz-node` header.
- Signature verification is a bounded, in-process parse-and-check; it does not
  fetch external collateral in the current implementation.

## Failure Handling

- A node whose envelope fails verification has the request retried on another
  path and its reputation reduced.
- Sustained failures, or a dispute resolved against the node, can lead to
  `slash_node`, which burns a portion of the node's stake once its reputation
  falls to the slashing threshold (`MIN_REPUTATION_BEFORE_SLASH`).

## Bootstrap Node Attestation

Bootstrap nodes participate in attestation like any other node once they run
on TEE-capable hardware. Their signing keys are provisioned inside the enclave
and never leave it; the deploy scripts enroll only the enclave's public
identity on chain.
