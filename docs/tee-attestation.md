# TEE Attestation

Wattz treats compute honesty as a first-class concern. Every response passes
through a TEE attestation check before its receipt is written on chain.

## Supported Attestation Types

### Intel SGX (DCAP)

- Quote is generated by the enclave running inside `node-runtime` and sent
  in `X-Wattz-Attestation`.
- Verifier: `packages/compute-verifier::tee::sgx`.
- Trust root: Intel's Provisioning Certification Enclave (PCE) chain,
  fetched at boot from the Intel Trusted Services (Trusted Attestation
  Services -- TAS) endpoint and pinned by SHA-256.

### AMD SEV-SNP

- Attestation report is produced by the SEV-SNP virtualization firmware.
- Verifier: `packages/compute-verifier::tee::sev`.
- Trust root: AMD Versioned Chip Endorsement Key (VCEK) chain.

### NVIDIA Confidential Computing (H100 CC / H200 CC)

- `nvattest` binary generates a per-response attestation blob.
- Verifier: `packages/compute-verifier::tee::nvidia_cc`.
- Trust root: NVIDIA NRAS root key, pinned by SHA-256.

Each response header is tagged with `X-Wattz-Attestation-Type` so the
verifier picks the correct path. Unknown types are rejected.

## Optional ZK Compute Proofs

For a subset of small models, the node may also attach a Risc0 or SP1 zkVM
proof of the entire inference computation. The verifier crate accepts these
via `packages/compute-verifier::zk::{risc0, sp1}`. When present, the
receipt PDA records the proof hash. The dispute path (`open_dispute` +
`resolve_dispute` on the Anchor program) can then require the ZK proof to
be re-verified in-band before resolving.

## Verification Timing

- The verifier runs on the gateway host, not on the client. Clients get
  only the summary claim: `is_verified: true|false` plus the trust root
  identifier.
- Verification adds ~5 ms for SGX/SEV. NVIDIA CC adds ~15 ms because of
  the additional NRAS certificate chain fetch (cached).

## Freshness Bounds

Every quote must satisfy:

- `quote.timestamp` within `ATTESTATION_MAX_AGE_SECS` (default 60) of now.
- `quote.nonce` equals the SHA-256 of `receipt.request_id`. The gateway
  supplies the nonce in the request header `X-Wattz-Nonce`.
- The signing key must chain to a trust root recorded in
  `packages/compute-verifier::trust_roots.rs`. The trust root set is
  updated by governance vote via the Anchor program's
  `update_trust_roots` instruction.

## Failure Handling

- A node whose quote fails verification is immediately marked degraded.
  A second failure within `NODE_QUARANTINE_WINDOW` (default 5 min)
  quarantines the node from the routing pool for 30 minutes.
- Repeated quarantines within 24 h trigger `slash_node`. Slashing burns
  a percentage of stake governed by `SLASH_RATE_BPS` (default 500 = 5 %).

## Bootstrap Node Attestation

Bootstrap nodes participate in attestation like any other node. Their
private keys live in `packages/bootstrap-nodes/attestation/`. The keys
are provisioned inside the enclave and never leave it -- the deploy
scripts only enroll the enclave's public identity on chain.
